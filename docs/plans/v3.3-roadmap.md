## A. Workflow Executor refactor + metering

### 痛點
v3.2 P2 已把 `meter_llm_call` 接到 3 個 chat 入口，但 **workflow LLM 呼叫沒接到**。
原因：`services/agent/app/core/workflow/executor.py` 的 `_exec_llm` / `_exec_image_understand` / `_exec_intent` 直接用 `AsyncOpenAI` client，繞過 BaseProvider。

### 4 PRs
- **A1 (PR 1)** — 整理 BaseProvider：補 `aembed` / `generate_image` / OpenAI-compat fallback；確保所有 provider 子類介面齊
- **A2 (PR 2)** — `WorkflowExecutor` 改 constructor 接 `session_factory`，所有 raw `AsyncOpenAI` 呼叫遷移到 `provider.chat()` / `provider.embed()`
- **A3 (PR 3)** — `_exec_*` node 包 `meter_llm_call`；workflow_run status 加 `"quota_exceeded"`，front-end 顯示專屬 chip
- **A4 (PR 4)** — workflow run history 顯示 token / cost；workflow node 加 `cost_usd` 欄位 (alembic 0003)

### Done when
- workflow run 跑一次 → `model_usage_logs` 對應寫 N 列（每個 llm node 一列）
- workspace cap = 0 → workflow run 直接停在 quota_exceeded、不再吃 token
- workflow run detail page 顯示 total cost

## D. Per-user quota + alert

### 痛點
v3.2 是 workspace 級。但實務上：
- 公司用 staffKM 給員工用，會想看「哪個員工花太多」
- 超額只是 429，沒有主動通知 admin

### 3 PRs
- **D1 (PR 5)** — alembic 0004：`user_quotas` 表 (user_id, workspace_id, monthly_token_cap, monthly_cost_cap_usd) + `check_quota` helper 同時檢 user / workspace 兩層
- **D2 (PR 6)** — alembic 0005：`quota_alerts` 表（threshold % / channel: email|slack|webhook / 觸發狀態）+ alert dispatch worker（背景跑、每 10 min 掃 80%/90%/100%）
- **D3 (PR 7)** — admin UI：每個 workspace detail 加「user usage」分頁 + alert 規則設定 modal；usage view 顯示「你個人 vs 整個 workspace」

### Done when
- 一個 user 達 cap → 同 workspace 其他 user 還能用
- 達 80% → email / Slack 收到通知（一次性，cap reset 才 re-arm）

## B. Observability deepening

### 痛點
v3.0 metrics 在、v3.1 audit 在，但 production debug 還是要拼 docker logs。
出 5xx 看不到 cross-service trace，慢 query 看不到 SQL 原文。

### 4 PRs
- **B1 (PR 8)** — OpenTelemetry：FastAPI / httpx / asyncpg auto-instrument，OTLP exporter → Tempo container；compose 加 tempo profile
- **B2 (PR 9)** — Loki + promtail：structlog JSON 經 docker driver → Loki；compose 加 loki profile；Grafana datasource provisioning
- **B3 (PR 10)** — Trace exemplar：v3.0 dashboards 接 `traceID` exemplar，從 metric 跳到對應 trace
- **B4 (PR 11)** — Grafana "Explore" 整合 dashboard：log → trace → metric 三向跳轉的 demo 頁

### Done when
- 開 Grafana → Explore → 看到 service map（gateway → agent → knowledge）
- 任一 5xx：從 log message 點 trace id → 跳 Tempo 看完整 span tree
- p95 latency dashboard 點 exemplar → 跳對應 slow trace

## C. RAG quality

### 痛點
MaxKB parity 是「能用」，但檢索準度沒實質提升。pure vector 對 acronym / 短字串差。

### 5 PRs
- **C1 (PR 12)** — alembic 0006：documents/paragraphs 加 tsvector column + GIN index；upload pipeline 補 tsvector 寫入
- **C2 (PR 13)** — Hybrid search：vector top-50 + BM25 top-50 → RRF (Reciprocal Rank Fusion) 融合 → top-20
- **C3 (PR 14)** — Re-ranker integration：Ollama 跑 `bge-reranker-v2-m3`（或 separate cross-encoder container），top-20 → top-5；前端設定頁可關
- **C4 (PR 15)** — Hit-test 視覺化升級：每段顯示 vector / BM25 / rerank 三組 score + 命中 highlight + RRF 算法說明
- **C5 (PR 16)** — Eval harness：固定 query set + ground truth → 跑 baseline / +BM25 / +rerank 三組對比，數字進 docs/perf

### Done when
- 同樣 query「LLM」在 baseline 拿不到（中文文件）、加 BM25 拿到、加 rerank 排第一
- hit-test page 看得到分數來源
- docs/perf/v3.3-rag-bench.md 紀錄三組 hit@5 / MRR 對比

| Risk | 緩解 |

| workflow refactor 動到既有 run，breaking | A1 先補介面、A2 灰度（feature flag）、A3 再正式接 metering |

## 為什麼這個順序

1. **A 先做** — 收 v3.2 尾，把 BaseProvider 介面打齊，後續 D / C 都會用到
2. **D 接著** — share v3.2 + A 的 metering / quota 基礎，最小 incremental
3. **B 第三** — cross-cutting infra，做完之後 C 的 RAG eval 也能用 trace 排查
4. **C 最後** — 純 feature、無依賴其他主題；可隨時砍掉留 v3.4 都不影響 v3.3 release
