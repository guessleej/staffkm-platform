# v3.6 Roadmap — Async resilience

> Status: planning（2026-05-18）
> Theme: 強化非同步任務的可靠性、idempotency、graceful shutdown
> 估時：2 週、4-5 PRs

---

## 痛點清單

| 痛點 | 影響 | 來源 |
|---|---|---|
| **Webhook 失敗就丟** | quota_alert / triggers 對外 webhook fail 只 log、不重試 | v3.3-D2 留尾 |
| **背景 task 不持久** | trigger_dispatcher / resume_worker / alert_worker 用 `asyncio.create_task`，process restart 就斷 | v2.x ~ v3.5 累積 |
| **無 idempotency** | 同一 chat / workflow run retry 會跑兩次（client 端 retry 友善）| 跨版本 |
| **K8s rolling update 不友善** | SIGTERM 沒等 in-flight workflow done 就被砍 | v2.0 helm 落地後 |

## 4-PR 拆解

### PR 1 — Webhook delivery retry + DLQ（P1, ~3 天）

**目標**：對外 webhook 改成 outbox pattern，失敗 exp backoff 重試，5 次後 DLQ。

**Schema**（alembic 0009）
```sql
CREATE TABLE webhook_outbox (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    workspace_id  UUID,                                -- nullable: system webhook
    url           TEXT NOT NULL,
    method        VARCHAR(8) NOT NULL DEFAULT 'POST',
    headers       JSONB,
    body          JSONB,
    status        VARCHAR(16) NOT NULL DEFAULT 'pending', -- pending/in_flight/delivered/failed
    attempts      INT NOT NULL DEFAULT 0,
    next_retry_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_error    TEXT,
    last_status_code INT,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    delivered_at  TIMESTAMPTZ,
    source_type   VARCHAR(32),                          -- quota_alert / trigger / ...
    source_id     UUID,
    CONSTRAINT chk_outbox_status CHECK (status IN ('pending','in_flight','delivered','failed'))
);
CREATE INDEX idx_outbox_due ON webhook_outbox(next_retry_at) WHERE status = 'pending';
CREATE INDEX idx_outbox_ws  ON webhook_outbox(workspace_id, status);
```

**Backoff schedule**：1m → 5m → 30m → 2h → 12h → DLQ（5 次）

**改動**
- `services/agent/app/core/quota_alert_worker.py` — webhook/slack 分支從直接 httpx.post → `enqueue_webhook(...)` 插 outbox
- `services/agent/app/core/webhook_dispatcher.py`（新）— 背景 loop，每 30s claim pending row → httpx.post → 成功 mark delivered / 失敗 schedule next retry
- 新 API `/admin/webhook-outbox`：list (含 status filter) + retry button + view failed
- 前端 `views/admin/WebhookOutboxView.vue`

### PR 2 — Background task durability + heartbeat（P1, ~3 天）

**目標**：背景 worker 加 heartbeat、graceful shutdown 等 in-flight done。

**Schema**（alembic 0010）
```sql
CREATE TABLE task_heartbeats (
    worker_name  VARCHAR(64) PRIMARY KEY,    -- trigger_dispatcher / resume_worker / alert_worker / webhook_dispatcher
    pid          INT,
    host         VARCHAR(128),
    started_at   TIMESTAMPTZ NOT NULL,
    last_beat    TIMESTAMPTZ NOT NULL,
    in_flight    INT NOT NULL DEFAULT 0      -- 目前在跑的 task 數
);
```

**改動**
- 所有 worker loop 加 `await _heartbeat(name)` 每 30s upsert
- 新 API `GET /admin/heartbeats` — admin 看 worker 健康
- Grafana panel：worker 心跳 freshness（last_beat 超過 5min 紅燈）
- 既有 `trigger_dispatcher` 等已用 `FOR UPDATE SKIP LOCKED` → durability 已 partial OK，主要補 heartbeat 觀測

### PR 3 — Idempotency keys（P2, ~3 天）

**目標**：client 傳 `Idempotency-Key` header，同 key 24h 內第二次 → 直接回原 response。

**Schema**（alembic 0011）
```sql
CREATE TABLE idempotency_keys (
    key            VARCHAR(128) NOT NULL,
    endpoint       VARCHAR(128) NOT NULL,
    workspace_id   UUID,
    response_json  JSONB,
    status_code    INT,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at     TIMESTAMPTZ NOT NULL DEFAULT (now() + interval '24 hours'),
    PRIMARY KEY (key, endpoint)
);
CREATE INDEX idx_idem_expire ON idempotency_keys(expires_at);
```

**實作**
- 新 middleware `app/middleware/idempotency.py`（agent service）
  - 只攔截 method=POST + 有 `Idempotency-Key` header
  - 同 key+endpoint 在 24h 內存在 → 直接回 cached response
  - 不存在 → 跑 handler、把 response capture 寫入
- SSE / streaming endpoint 跳過（response body 無法 cache）
- 文件 `docs/dev/idempotency.md` + Postman example

### PR 4 — Graceful shutdown + DR drill doc（P2, ~2 天）

**目標**：SIGTERM 後等 in-flight task 收尾才退；補 DR 文件。

**實作**
- agent service lifespan：
  - shutdown handler 標 `_shutting_down = True`
  - 等 `event_trigger_runs WHERE status='running'` 全 done 或 30s timeout
  - alert_worker / resume_worker / webhook_dispatcher 全部 cancel
  - 最後 close DB pool
- compose `agent` service `stop_grace_period: 60s`
- 新文件 `docs/deploy/dr-drill.md`：
  - K8s rolling update 場景：rollout restart → 預期行為 / 驗證步驟
  - PostgreSQL failover：primary 掛 / replica promote
  - Redis 重啟：CAPTCHA fallback 行為
  - Backup restore：從 PG dump / MinIO snapshot 復原
