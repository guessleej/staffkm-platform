## 4 PR 拆解

### PR 1 — Non-LLM workflow node metering（P1, ~3 天）

**痛點**：v3.3-A 只接 4 個 LLM-based node，剩 4 個沒接：
- `_exec_image_generate` — per-image pricing（dall-e-3: $0.04/image standard / $0.08/image HD）
- `_exec_stt` — per-second audio pricing（whisper: $0.006/min）
- `_exec_tts` — per-character pricing（tts-1: $15/1M chars）
- `_exec_reranker` — per-call / per-token pricing（cohere rerank: $1/1k searches）

**設計**
- alembic `0006_media_pricing`：`ai_models` 加：
  - `price_per_image_usd` NUMERIC(10, 6) — image-gen
  - `price_per_second_usd` NUMERIC(10, 6) — STT
  - `price_per_1k_chars_usd` NUMERIC(10, 6) — TTS
  - `price_per_call_usd` NUMERIC(10, 6) — reranker
- `model_pricing.py` 加 seed：dall-e-3 / whisper-1 / tts-1 / bge-reranker / cohere-rerank
- 擴充 `model_usage_logs`：加 `unit_type` (string: token/image/second/char/call) + `unit_count` (numeric)；nullable backward compat
- `metering.py` 加 `meter_media_call()` context manager（pre check_quota，post 算 cost、寫 model_usage_logs，但用 unit-based 而非 token-based）
- 4 個 `_exec_*` 改用 `meter_media_call`

### PR 2 — Email SMTP dispatch（P2, ~2 天）

**痛點**：v3.3-D2 alert worker 對 email channel 只 log

**設計**
- 新檔 `services/agent/app/core/email.py`：thin wrapper 用 `aiosmtplib`
  - SMTP host / port / user / password / from_addr 從 settings（.env）
  - 沒設 → `send()` 回 warning + log skip（dev 友善）
- `quota_alert_worker.py` 的 `_dispatch` email 分支改成 `await send_email(to=alert["target"], subject="...", body="...")`
- `requirements.txt` 加 `aiosmtplib==3.0.2`
- compose `agent` service environment 加：
  ```
  SMTP_HOST: ${SMTP_HOST:-}
  SMTP_PORT: ${SMTP_PORT:-587}
  SMTP_USER: ${SMTP_USER:-}
  SMTP_PASSWORD: ${SMTP_PASSWORD:-}
  SMTP_FROM:     ${SMTP_FROM:-noreply@staffkm.local}
  ```
- `.env.example` 加註解版

### PR 3 — Cross-encoder reranker container（P2, ~3 天）

**痛點**：v3.3-C 的 Ollama reranker fallback 是用 embedding cosine，效果差

**設計**
- 新 service `services/reranker/`（FastAPI thin wrapper）：
  - 跑 `sentence-transformers` + `bge-reranker-v2-m3`
  - 一個 endpoint：`POST /rerank` body `{query, documents: [str], top_n}`，回 `{indices, scores}`
  - 模型 cold start ~5s（lazy load）
- compose 加 service（profile `reranker`，預設 disabled，可選啟用）：
  ```yaml
  reranker:
    image: staffkm-reranker:latest
    profiles: ["reranker"]
    ports: ["8010:8000"]
    environment:
      MODEL_NAME: BAAI/bge-reranker-v2-m3
  ```
- `core/reranker.py` 加 `_rerank_cross_encoder` dispatch：
  ```python
  elif reranker_type == "cross_encoder":
      return await _rerank_cross_encoder(query, documents, config, top_n)
  ```
  config 帶 `endpoint: http://reranker:8000`
- 文件 `docs/dev/reranker.md`：opt-in 方式 / model 選擇 / 大概延遲

### PR 4 — RAG eval annotation + CI bench job（P3, ~2 天）

**痛點**：v3.3-C5 dataset 的 `expected_paragraph_ids` 都是空的，eval 跑不起來

**設計**
- 寫 `tools/eval/seed_eval_kb.py`：
  - 從 `tools/eval/seed_corpus/` 讀 markdown 檔（10 篇 HR / 政策範例文件，新建）
  - 上傳到指定 workspace 的 eval KB（用 `/knowledge/kbs/{id}/documents` API）
  - 文件分段後印出每段 paragraph_id 給 review，然後手動填回 `rag_eval_dataset.json`
- 真實標註 10 條 query 的 expected_paragraph_ids（從 seed 文件挑明顯該命中的段落 ID；以註解標示「dummy ID, 真實標註需配對實際分段」）
- 新 GH Action `.github/workflows/rag-eval.yml`：
  - 每週日 02:00 UTC 跑
  - 啟 docker compose、seed eval KB、跑 `rag_eval.py`
  - 把結果 commit 進 `docs/perf/v3.3-rag-bench-history.md` (append)
  - 設 threshold：若 hybrid hit@5 < 0.6 fail CI
- docs 補：人工標註流程 + CI 怎麼跑
