# v3.1 Roadmap — 收尾 v3.0 留下的尾巴

> Status: planning（2026-05-18）
> Theme: **Technical debt cleanup**（純收尾、無新功能）
> 估時：1.5 週、4 PRs

---

## 為什麼是這個主題

v3.0 把基礎打穩了，但留了 3 條技術債明確標為「未來 PR」：

1. **Alembic baseline migration**（v3 P4 跳過）
   - 現在 schema 演化靠各 service `_*_BOOTSTRAP_DDL` list + `ALTER ... IF NOT EXISTS`
   - 多 instance 同時起會 race（雖然 idempotent，但 DDL lock 競爭）
   - 沒有版本鏈 → 看不出當前 schema 在哪個版本、不能 downgrade
2. **`LEGACY_URL_MODE` 仍預設 `rewrite`**
   - v3.0 預告「v3.x 改 `410`、v4.0 移除」
   - 拖越久殘留 caller 越難挖、bridge code 越難拔
3. **Audit log `_record()` helper 沒人 call**
   - PR #177 寫好了 API + helper，但沒在任何寫操作埋點
   - 結果：admin 進 `/admin/audit-logs` 永遠空表

v3.1 把這三條全收。

## 4-PR 拆解

### PR 1 — Alembic baseline（P1, ~3 天）

**目標**：把現有 schema 「凍結」成 Alembic 版本 `0001_baseline`，新 schema 改走 `alembic revision`。

**範圍**
- 加 `services/{auth,agent,knowledge,chat,integration}/alembic/` + `alembic.ini`
- 用 `alembic revision --autogenerate` 對著 running DB 產 baseline
- 各 service `main.py` lifespan 開頭跑 `alembic upgrade head`（取代 `_run_*_bootstrap_ddl()`）
- 保留 `bootstrap_ddl.py` 但加 deprecation comment：「v3.1 之後新 schema 走 alembic」
- DDL race fix：用 `pg_advisory_lock(hashtext('staffkm_migrate'))` 包 upgrade，多 instance 安全

**Risk / 緩解**
- 線上 DB 已有資料，autogenerate 可能誤判 → 手動 review diff、必要時改成 `op.execute("...IF NOT EXISTS")`
- partitioned table（audit_logs）autogenerate 不支援 → baseline 用 raw SQL

**Done when**
- `docker compose up` cold start：5 個 service 各跑 `alembic upgrade head` 成功
- 新 PR 加欄位 demo：`alembic revision -m "add foo to bar"` → 自動產 migration → upgrade pass

### PR 2 — Audit `_record()` 接線（P2, ~2 天）

**目標**：在 6 類關鍵寫操作埋 audit log，admin 後台看得到真實軌跡。

**埋點清單**
| Resource | Actions |
|---|---|
| `application` | create / update / delete / publish |
| `kb` | create / delete / sync_web |
| `api_key` | create / revoke |
| `user` | create / role_change / delete |
| `template` | install_from_marketplace / publish |
| `workflow_run` | trigger / cancel |

**Pattern**
```python
from app.api.audit import _record
await _record(
    session,
    workspace_id=ctx.workspace_id,
    actor_user_id=ctx.user_id,
    actor_username=ctx.username,
    action="create",
    entity_type="application",
    entity_id=app.id,
    entity_label=app.name,
    detail={"template_id": tpl_id} if tpl_id else None,
    ip_address=request.client.host,
    user_agent=request.headers.get("user-agent"),
)
```

**Done when**
- 手動跑 6 個操作 → 進 `/admin/audit-logs` 看得到 6 列
- E2E spec `06-audit.spec.ts` 建 app → 驗 audit row

### PR 3 — Legacy URL sunset Phase 1（P2, ~3 天）

**目標**：把所有 caller 改成打 workspace-scoped URL，然後 default 翻成 `410`。

**Step 1 — Caller 改寫（不改 default）**
- **Gateway** `_generic_proxy.py`：proxy target 從 `{base}/api/v1/{prefix}` 改 `{base}/api/v1/workspace/{ws}/{prefix}`
  - 從 request header `X-Workspace-ID`（gateway auth middleware 已注入）拿 ws
- **Frontend** `apps/web/src/api/*.ts`：grep `'/api/v1/'` 全表盤點
  - 已 workspace-scoped 的 endpoint 沒事
  - 還在打 legacy 的（tools / skills / data-sources / folders / triggers / mcp / memories / app-templates / audit-logs）改打 `/api/v1/workspace/${wsId}/...`
  - 拿 `useWorkspaceStore().currentWsId` 注入

**Step 2 — 翻 default**
- `LegacyURLBridge.__init__`：`self.mode = os.environ.get("LEGACY_URL_MODE", "410").lower()`
- 部署設 `LEGACY_URL_MODE=410`，跑 1 週看有沒有 5xx → 沒有就 merge

**Done when**
- dev compose 設 `LEGACY_URL_MODE=410`、所有 E2E pass
- production 灰度 1 週 0 unexpected 410
