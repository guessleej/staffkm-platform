# v3.7 Roadmap — Cost attribution + LLM-as-judge + Slow query trace

> Status: planning（2026-05-18）
> Theme: 從 quota 拓展到 **per-conversation cost**、評估從 hit@5 升到 **LLM 評**、observability 補 slow query
> 估時：2 週、4-5 PRs

---

## 痛點清單

| 痛點 | 影響 | 來源 |
|---|---|---|
| **沒有 per-conversation cost** | admin 看不到「這次對話花多少」、無法跟 user 收費 | v3.2-P2 metering 只到 workspace/user 級 |
| **eval 只看 hit@5 / MRR** | 答案品質沒有「合不合理」評分 | v3.3-C5 limitation |
| **chat / workflow / hit-test / embed 不分流** | dashboard 只看 workspace 總額，無法判別哪個 feature 花錢 | v3.2-P3 dashboard limitation |
| **慢 SQL 看不到** | 偶發 500+ms query 沒落地、debug 要從 OTel 拼 | v3.3-B1 OTel auto-instrument 自動但無高亮 |

---

## 4-PR 拆解

### PR 1 — Cost attribution per conversation（P1, ~3 天）

**Schema**（alembic 0012）
- `chat_messages` 加 `cost_usd NUMERIC(10, 6)` + `tokens_used INT`
- `model_usage_logs` 加 `conversation_id UUID`（nullable）

**改動**
- `metering.py` `meter_llm_call` 加 `conversation_id` kwarg → 寫入 model_usage_logs
- chat_stream / applications.chat / public.chat 3 處從 request body / context 取 conversation_id 傳進
- chat_message 寫入時聚合 cost / tokens 回填（從 model_usage_logs WHERE conversation_id=...）
- 前端 chat UI 每則 assistant message 顯示小字 `💰 $0.0123 · 🔤 1.2k`

### PR 2 — Cost-by-feature dashboard（P2, ~2 天）

**Schema**：不動。利用 `model_usage_logs` 既有的 `application_id` + `unit_type` + `conversation_id` 已能 group。
- 加 `feature` 推斷邏輯：application_id 對應 application.type（chat / workflow / 等）
  - 簡化：直接用 source path 推斷（chat_stream → 'chat'; trigger_dispatcher → 'workflow'; hit-test 沒進 metering 故略；embed 沒進 metering）
- 加 `model_usage_logs.feature VARCHAR(16) nullable`（alembic 0013，看是否合併到 0012）
- 寫入時直接帶 feature

**Grafana**
- 新 dashboard `v3-cost-by-feature.json`：
  - bar：本月 cost by feature
  - timeseries：daily cost by feature
  - table：top conversations by cost
  - table：top users by cost

### PR 3 — LLM-as-judge eval（P2, ~3 天）

**改動**
- `tools/eval/rag_eval.py` 加 `--judge-model` 參數：
  - 對每 query：拿 retrieval results → feed 給 judge LLM 評「retrieved 內容是否能回答 query (0-5)」
  - 計算 average score per mode
- 新 dataset 欄位 `judge_criteria`（每 query 額外給 judge 的 rubric）
- README 補：如何選 judge model（不能跟測試 model 同個）
- bench history 多一欄 `judge_avg`

### PR 4 — Slow query trace（P2, ~2 天）

**做法**
- 新 middleware `services/agent/app/middleware/slow_query.py`：用 SQLAlchemy event listener 監聽所有 query
- 超過 500ms：
  - 結構化 log（含完整 SQL + bind params + duration）
  - 加進當前 OTel span 的 tag (`db.slow=true`、`db.duration_ms=N`)
- compose 加 env `SLOW_QUERY_THRESHOLD_MS` 預設 500
- Grafana `v3-observability-demo.json` 加 panel：「24h slow queries by service」（從 Loki 撈 log）
