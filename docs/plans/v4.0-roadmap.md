# v4.0 Roadmap — 大改

> Status: planning（2026-05-18）
> Theme: **拔技術債 + distributed task queue + multi-region readiness**
> 估時：3-4 週、~8 PRs
> SemVer：major — 預期有 breaking changes，operator 必須讀 migration

---

## 為什麼是 v4.0

v3.x 累積 8 個 minor，技術債到了該動的時候：
- v3.1 / v3.6 標 LegacyURLBridge 「v4.0 移除」
- v3.1 標 bootstrap_ddl 「v4.0 移除」
- v3.6 標 distributed task queue 「v4.0 大改」
- DR drill 4 場景中 multi-region 還沒落地

v4.0 一次做完，避免拖到 v4.x。

## 8-PR 拆解（依風險由低到高）

### PR 1 — 拔 bootstrap_ddl.py（低風險, ~2 天）
v3.1 加 alembic baseline 後，bootstrap_ddl 一直是 backstop（idempotent ALTER ADD COLUMN）。
v4.0 假設所有 deploy 都已到 v3.1+ → 移除：
- 確認 alembic 0001 → 0014 鏈完整（每個 schema 變更都有對應 revision）
- 刪 `services/agent/app/bootstrap_ddl.py`
- 刪各 service 的 `_*_BOOTSTRAP_DDL` list
- lifespan 只剩 `await run_alembic_upgrade()`
- 文件補：v4.0 開始 schema 變更**唯一管道**是 alembic revision

### PR 2 — 拔 LegacyURLBridge（低-中風險, ~2 天）
v3.1 default 已 `410`；v3.6 升级時可選 rewrite 回滾。v4.0 完全移除。
- 刪 `services/agent/app/middleware/legacy_bridge.py`
- 刪 `app.add_middleware(LegacyURLBridge)`
- 刪相關 env `LEGACY_URL_MODE`
- 確認 gateway proxy 還能 forward 到 workspace-scoped agent endpoint（v3.1 已對齊）
- 文件補：升 v4.0 前**必須**確認沒人打 `/api/v1/{prefix}/...` legacy URL
- E2E test 加：故意打 legacy URL → 預期 404（之前是 410）

### PR 3 — Distributed task queue 基礎（高風險, ~5 天）
取代 5 個 in-process worker。選 **Arq**（async-native + Redis-backed，比 Celery 輕）：
- 新 service `services/worker/`：FastAPI-less，純 arq worker process
- 5 個既有 worker loop 改寫成 arq job：
  - trigger_worker（schedule scan）
  - trigger_dispatcher（claim queued runs）
  - resume_worker（approved 續跑）
  - alert_worker（quota evaluate）
  - webhook_dispatcher
- agent service lifespan 不再啟動 worker，改 enqueue 到 redis
- compose 加 `worker` service（profile core）
- heartbeats / task durability 改用 arq 內建（保留 task_heartbeats 表給歷史）
- 文件 `docs/dev/distributed-workers.md`：架構圖 + 為什麼選 arq

### PR 4 — Distributed task queue migration helpers（高風險, ~3 天）
拆 PR 3 太大會難 review，這 PR 補：
- 平行運行模式（feature flag `WORKER_BACKEND=inprocess|arq`）
- Helm chart 加 worker deployment（separate replicas）
- prometheus exporter for arq queue depth
- Grafana panel：queue depth / job latency / failure rate

### PR 5 — Multi-region readiness — PG streaming replication（高風險, ~3 天）
- Helm chart 加 PG replica StatefulSet（read-only follower）
- compose 加 `postgres-replica` service（profile multi-region）
- 文件 `docs/deploy/multi-region.md`：
  - active-passive 設定（手動 failover）
  - active-active 限制（v4.0 不支援，需 conflict resolution）
  - RTO/RPO 目標

### PR 6 — Multi-region readiness — read replica 路由（中風險, ~2 天）
- `staffkm_core/utils/database.py` 加 `init_db(write_url, read_url=None)`
- get_session() 不變；新 `get_read_session()` 走 replica（讀重的 endpoint 用）
- 標一批 read-only endpoint：billing report / audit logs / dashboard query
- 文件範例

### PR 7 — Multi-region readiness — MinIO replication（中風險, ~2 天）
- compose 加 secondary MinIO（profile multi-region）
- 設 bucket replication rule
- 文件 + DR drill 補

### PR 8 — v4.0 changelog + release notes + 升級 wizard（~1 天）
- CHANGELOG 詳細 breaking changes list
- 新文件 `docs/upgrade/v3-to-v4.md`（step-by-step migration）
- 標 v4.0 為 major release（前所未有的 breaking changes 量）

## 跨切議題

### Backward compat
- v3.8 → v4.0：alembic chain 必須跑完才能升
- v3.1 之前升：不支援，必須先升到 v3.1+
- gateway 不向後相容 LegacyURLBridge 移除（custom client 必須帶 X-Workspace-ID 或打 workspace-scoped URL）

### Rollback 策略
- DB schema：alembic downgrade 可走（每 migration 都有 downgrade）
- Code rollback：image 換回 v3.8（前提 schema 仍兼容）
- LegacyURLBridge：v4.0 移除後**無法 rollback** — 必須先驗證 0 caller

### 何時 release
- 建議跑 1 個月 staging 才正式 v4.0.0
- PR 8 內加 "v4.0.0-rc1" 預發版本標籤建議

## 升級路徑

```
v3.8.0 ──(此 milestone, breaking)── v4.0.0
            │
            ├─ alembic 鏈走完（必須 v3.1+ 才能升）
            ├─ bootstrap_ddl.py 刪除（schema 只走 alembic）
            ├─ LegacyURLBridge 刪除（caller 必須帶 X-Workspace-ID）
            ├─ Worker 全部搬到 arq / redis（in-process 模式 deprecated）
            └─ Helm chart 支援 PG replica + MinIO secondary
```
