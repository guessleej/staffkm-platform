# v3.3 RAG eval — method & baseline

> v3.3 在 `vectorstore.py` 落地 CJK tsvector + RRF hybrid，並把 reranker 接上 `hit_test`。
> 本檔記錄如何用 `tools/eval/rag_eval.py` 量化 retrieval 品質、追蹤 baseline。

## 為何 eval

C1+C2 的 hybrid + RRF、C3 的 reranker，都是「直覺上應該變好」的改動。
沒有 hit@5 / MRR 數據，就無從判斷：

- hybrid 真的比 vector-only 好嗎？對 zh-TW corpus 改善多少？
- reranker (Cohere 雲端 vs Ollama bge fallback) 帶來的增益值不值得 latency？
- 後續 chunking 策略改動有沒有 regression？

## 怎麼建 eval KB

1. 在 staffKM 建一個 KB（命名 `staffkm-eval-kb`），上 10-30 篇 HR / policy / 流程 zh-TW 文件
   - 內容廣度蓋住 `rag_eval_dataset.json` 那 10 條 query 的 topic
2. 對每條 query，到 `/knowledge/<kb>/paragraphs` 找出「人類認為正確」的 paragraph，
   把這些 paragraph 的 UUID 填回 `rag_eval_dataset.json` 的 `expected_paragraph_ids`
3. commit 到 repo（dataset 體積很小，version control 友善）

> 沒填 `expected_paragraph_ids` 的 query 會被 runner skip 並計數，不影響其他 query 統計。

## 怎麼跑

```bash
# 環境變數（按需設定）
export COHERE_API_KEY=...
export OLLAMA_BASE_URL=http://localhost:11434

# 三種 mode + 兩種 rerank
python tools/eval/rag_eval.py \
  --base http://localhost \
  --token "$JWT" \
  --kb-id 11111111-2222-3333-4444-555555555555 \
  --dataset tools/eval/rag_eval_dataset.json \
  --modes vector hybrid fts \
  --rerank none cohere ollama \
  --output docs/perf/v3.3-rag-bench-results.md
```

JWT 取得：到 `/login` 拿一個 workspace member token；或直接從 browser localStorage 撈 `access_token`。

## 預期 baseline（placeholder）

| mode+rerank   | hit@5 | MRR  |
|---------------|-------|------|
| vector+none   | ~0.60 | ~0.45 |
| fts+none      | ~0.55 | ~0.42 |
| hybrid+none   | ~0.75 | ~0.58 |
| hybrid+ollama | ~0.78 | ~0.62 |
| hybrid+cohere | ~0.85 | ~0.72 |

> 上表為「直覺預期」placeholder，PR 內不實跑（沒 real KB / 沒 JWT）。
> 第一次跑完後請覆蓋此區塊的數字，並把 `docs/perf/v3.3-rag-bench-results.md` commit 進 repo。

## v3.4-P4 標註完成 + CI 跑

### 新增資源
- `tools/eval/seed_corpus/` — 10 篇 zh-TW HR / policy 文件（leave / overtime / expense /
  onboarding / remote / equipment / IT security / business trip / performance / resignation），
  每篇 4 段獨立標題，便於 anchor 標註。
- `tools/eval/seed_eval_kb.py` — 把 seed corpus 上傳到目標 workspace，poll 等
  embedding 完成，輸出 `seeded_paragraphs.json`（含每段 UUID + title + content 摘要）。
- dataset schema 加 `expected_corpus_anchors`：
  ```json
  { "file": "01-leave-policy.md", "title_contains": "年假" }
  ```
  10 條 query 全部已標註 anchors（共 ~22 個 anchor），實際 paragraph_id 在 runner 端
  解析。
- `rag_eval.py` 加 `--seeded path/to/seeded_paragraphs.json` 參數，啟動時 resolve
  anchors → expected_paragraph_ids，對不到 file / title 會 log warning 並略過該 anchor。

### 人工確認流程
1. 跑 `seed_eval_kb.py` → 拿 `seeded_paragraphs.json`
2. 開檔人眼掃過：確認每篇 4 段 title / content 摘要看起來合理（chunker 沒切爛）
3. 跑 `rag_eval.py --seeded ...`，留意 stdout `[anchor-miss]` 行，若有需調整 anchors
4. 結果寫進 `docs/perf/v3.3-rag-bench-results.md`（單次）或 history doc（CI 累積）

### CI 跑法
- `.github/workflows/rag-eval.yml` 每週日 02:00 UTC 自動跑，亦支援 manual dispatch
- 流程：docker compose up → login admin → seed KB → run eval (vector / hybrid / fts) →
  append history → threshold gate (`hybrid+none` hit@5 >= 0.5) → commit history
- 不跑 reranker container（cold start 太久；待 P5+）
- 首次 enable 前確認 runner 記憶體（>=8GB）

## 注意事項

- Ollama reranker（C3.5）目前在 `core/reranker.py` 採「先試 `/api/rerank`、失敗 fallback embedding cosine」策略，
  cosine fallback 效果有限，預期 hit@5 / MRR 增益只略勝 hybrid+none；v3.4 會換 cross-encoder。
- 樣本只有 10 條，方差大；正式追蹤建議擴到 50+ 條並加 95% CI。
- runner 不會自動清 KB / 不會自動 reindex；如要對比 chunking 變更，需手動換 KB 或重 build embedding。

## v3.7 P3：LLM-as-judge

`rag_eval.py` 加 `--judge-model` 參數，啟用 LLM 評分（0-5）。

```bash
python tools/eval/rag_eval.py \
    --base http://localhost --token $TOKEN --kb-id $KB \
    --dataset tools/eval/rag_eval_dataset.json --seeded /tmp/seeded.json \
    --modes hybrid \
    --judge-model gpt-4o-mini \
    --judge-base https://api.openai.com/v1 \
    --judge-key $OPENAI_API_KEY \
    --output /tmp/bench.md
```

輸出 markdown table 多一欄 `judge_avg`（0-5 平均分）。

**選 judge 注意**：
- 不要用跟 retrieval pipeline 同一個 LLM model 評（會偏袒）
- gpt-4o-mini / gemini-1.5-flash 都夠用、便宜
- 也可選 Claude / 自架 Llama-70B；越強越穩

**dataset 加 `judge_criteria`**（可選）：
```json
{
  "query": "員工請假上限",
  "judge_criteria": "需明確涵蓋年假/事假/病假三種類型的天數"
}
```
會 append 進 judge prompt，給更精確的 rubric。

不提供 `--judge-model` → 維持只跑 hit@5 / MRR（向後相容）。
