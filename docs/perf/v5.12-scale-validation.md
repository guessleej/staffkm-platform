# 規模驗證 — 10 萬段落 + 多 KB + 並發查詢（v5.12）

> 目的：把 RAG 召回從「小語料單一 KB」推到接近 production 量級，拿**真實測得**的數字
> （非「設計上應該可以」）。harness：`tools/perf/scale_validation.py`（可重跑）。

## TL;DR

- **100,000 段落 / 10 個 KB / 1024 維** 真實載入 pgvector，建 ivfflat 索引並做並發查詢，**全部實測**。
- 載入 **24.6s**、ivfflat 建索引 **12.2s**（lists=316）。
- 單查 p50 **4.5ms（probes=1）→ 35ms（probes=10）**：**probes 在 10 萬量級是 ~6× 延遲槓桿**（小語料時可忽略）。
- 32 並發：throughput **151 qps**、p50 **192ms**、p99 **380ms**（單機 laptop 容器）。
- **KB 隔離在 10 萬量級成立**（查 KB-A 不外洩 KB-B 段落）。

## 環境（誠實標註：單機，非 production 硬體）

| 項 | 值 |
|---|---|
| 主機 | Apple M2，12 vCPU / 32 GB（Docker Desktop，容器上限 ~7.65 GB） |
| DB | `pgvector/pgvector:pg16`，`--shm-size=512m`，`maintenance_work_mem=256MB`（建索引用） |
| 資料 | 100k 段落、10 KB（各 ~10k）、1 doc/KB；embedding =**隨機 1024 維** |
| 索引 | ivfflat `vector_cosine_ops`，lists=316（≈√100k），**載入後**才建 |
| 查詢 | 鏡像 `vectorstore.hybrid_search` 的 vector 模式 SQL（JOIN paragraphs+documents、`<=>` cosine、`SET LOCAL ivfflat.probes`、top_k=5） |

## 實測數字（`tools/perf/scale_validation.py --paragraphs 100000 --kbs 10 --dim 1024 --queries 500 --concurrency 32`）

### 載入 / 建索引
| 項 | 秒 |
|---|---|
| 載入 100k 段落 + 100k 向量（COPY） | **24.6** |
| 建 ivfflat 索引（lists=316） | **12.2** |

### 單查延遲（500 次，序列）
| probes | mean | p50 | p95 | p99 | max |
|---|---|---|---|---|---|
| **1**（pgvector 預設） | 5.56 ms | 4.53 ms | 11.96 ms | 15.62 ms | 37.68 ms |
| **10**（staffKM `IVFFLAT_PROBES`，v5.11.4） | 34.58 ms | 35.05 ms | 45.02 ms | 49.85 ms | 52.43 ms |

### 並發（32 worker × 500 查詢，probes=10）
| metric | 值 |
|---|---|
| throughput | **151.3 qps** |
| wall | 3.31 s |
| p50 / p95 / p99 | 192 / 336 / 380 ms |

### KB 隔離
- 查 3 個不同 KB，每次回傳結果**全部屬於該 KB** → **isolation_ok = true**。

## 關鍵發現（可行動）

1. **probes 是 10 萬量級的真實延遲槓桿（~6×）**。小語料（v5.11.4 調校時）probes=1→10 延遲幾乎無差，所以當時為**召回**設 `IVFFLAT_PROBES=10` 無痛。但 100k 時 p50 從 4.5ms → 35ms。
   - 仍在可接受範圍（p99 < 50ms 單查），但語料再上一個量級（百萬）應把 `IVFFLAT_PROBES` 納入**召回 vs 延遲預算**一起調，而非定值。
   - ⚠ 本 harness 用**隨機向量** → **recall@k 不具意義**（隨機向量無語意鄰近結構）；recall 的調校仍須用真 embedding（見 `tools/eval/graphrag_ab.py`）。本報告的延遲 / 吞吐 / 索引 / 隔離數字才是有效結論。

2. **並發下延遲放大**：單查 probes=10 p50 35ms，但 32 並發時 p50 192ms（~5.5×）。單機 laptop 容器 CPU-bound + pool 競爭所致；production 多核 + 連線池調校會更好。**151 qps 是單機 pgvector 的 baseline 地板**，非 SLA。

3. **建索引吃 `maintenance_work_mem`**：100k×1024 ivfflat 需 ~86MB，預設 64MB 會 `ProgramLimitExceeded`。→ **大庫重建索引（含 embedding 熱換 reindex）前必須拉高 `maintenance_work_mem`**。已寫進 harness + 此處備忘。

4. **KB 隔離在量級下成立**：`WHERE pe.kb_id = :kb` + 每 KB 各自候選，10 KB 混存無外洩。

## 限制 / 不宣稱

- 單機容器，**絕對數字是地板/相對值**，非 production SLA。
- 隨機向量 → **不含 recall 結論**（只測 latency/throughput/index/isolation）。
- 未測：跨 KB 聯合查詢、FTS+hybrid 在 100k 的相對成本（vector-only 為主）、ivfflat vs HNSW。
